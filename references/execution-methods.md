### 格式规范

每条修改包含：


## #28 修改前后逐段比对

### 方法

1. 提取当前文档全文到txt（逐段编号）
2. 对照修改清单逐段检查执行状态
3. 标注执行结果

### 标注符号

| 符号 | 含义 |
|:----:|------|
| ✓ | 已修改完毕 |
| ✗ | 待修改 |
| • | 无需修改 |
| ⚠ | 修改有偏差需复查 |

### 输出格式

```
P014 ✓ 已修改
P016 ✗ 待修改（归属未修正）
P020 • 无需修改
P027 ✓ 已修改
...
```

### 操作建议

- 从末尾向前操作避免段落索引漂移
- 修改一处的段落索引变化可能影响后续比对，建议分批次比对
- 每批次<10处修改后重新提取全文

## #30 跨段语义一致性审查

### 背景

自动化审查框架的所有检查项都是单段模式匹配（正则/关键词/数据点扫描），对**跨段语义矛盾**天然盲视。例如：一个段落定义了"等"（列举未尽），另一个段落用精确数量词（"五项"）指代该列举，两个段落跨越章节边界后，在缺乏语义理解能力的自动化审查中必然漏过。此类问题需要专门的跨段比对方法。

### 检测模式

#### 模式A："等"与精确数量词冲突

| 定义段（定义方） | 指代段（引用方） | 矛盾 |
|:--|:--|:--|
| `包含A、B、C、D、E**等**在内的……` | `上述**五项**交付文件` | "等"=列举未尽 → 不应写死为"五项" |
| `主要功能包括X、Y、Z**等**` | `以上**三类**功能` | "等"=可能更多 → 不应锁定为"三类" |
| `涵盖……等多个方面` | `上述**七个**方面` | "多"=不确定数量 → 不应写死为"七个" |

**修复方案**：删除精确数量词 → "上述交付文件" / "以上功能" / "上述方面"。

#### 模式B：列举数量与宣称数量不一致

| 定义段 | 指代段 | 矛盾 |
|:--|:--|:--|
| `系统包含十大核心模块`（随后列出8个） | "十大模块"重复出现 | 宣称10但列举8 |
| `四大设计原则如下……`（列出5条） | "上述四大原则" | 宣称4但列举5 |

**修复方案**：使宣称数量与实际数量一致（修正宣称数或修正列举数）。

#### 模式C：中文数量词全量扫描

全文提取所有中文数量词（如"三项""五类""七个""八条""十二套"），逐条与所指数量的实际元素比对。

| 中文数量词 | 正则模式 |
|:--|:--|
| 基数+量词 | `[一两三四五六七八九十百千]+(项|类|个|条|套|种|层|级|步|环|方面|维度|模块|环节)` |

### 操作流程

```
1. 全文扫描所有"等"字列举段落（定义段列表）
2. 全文扫描所有"上述N项/类/个"指代句式（指代段列表）
3. 交叉比对：指代段是否指向定义段？定义段是否含"等"？
4. 对冲突项输出修复建议
5. 可选：启动模式C全量中文数量词审计
```

### P0级判定条件

- 定义段含"等"且指代段含精确数字 → **P1**（措辞矛盾，读者可自行推断但仍需修正）
- 宣称数量与列举数量不一致 → **P0**（硬伤，可直接导致审稿人质疑严谨性）

### 输出格式

```
跨段语义一致性审查报告
========================
定义段:
  L84: 包含质量证明书、材质单、检测报告、产品合格证、监检证**等** → 含"等"，数量不确定

指代段:
  L105: 上述**五项**交付文件 → 与L84"等"冲突

修复: L105 删除"五项" → "上述交付文件"
```

### 经验笔记

- 此类问题在交付包去重修复中常见：合并重复罗列时，为简洁而写出精确数量，忽略了原定义段的"等"字
- 审查工具是模式匹配而非语义理解，跨段语义矛盾不能用正则一步检出，必须建立专门的比对流程
- 中文"等"的语用含义有二：①列举未尽（"苹果、橘子等水果"）②列举煞尾（如公文用语），论文中通常为含义①
