## #28 修改前后逐段比对

### 方法

1. 提取当前文档全文到txt（逐段编号）
2. 对照修改清单逐段检查执行状态
3. 标注执行结果

### 标注符号

| 符号 | 含义 |
|:----:|------|
| ✓ | 已修改完毕 |
| ✗ | 待修改 |
| • | 无需修改 |
| ⚠ | 修改有偏差需复查 |

### 输出格式

```
P014 ✓ 已修改
P016 ✗ 待修改（归属未修正）
P020 • 无需修改
P027 ✓ 已修改
...
```

### 操作建议

- 从末尾向前操作避免段落索引漂移
- 修改一处的段落索引变化可能影响后续比对，建议分批次比对
- 每批次<10处修改后重新提取全文

## #30 跨段语义一致性审查

### 背景

自动化审查框架的所有检查项都是单段模式匹配（正则/关键词/数据点扫描），对**跨段语义矛盾**天然盲视。例如：一个段落定义了"等"（列举未尽），另一个段落用精确数量词（"五项"）指代该列举，两个段落跨越章节边界后，在缺乏语义理解能力的自动化审查中必然漏过。此类问题需要专门的跨段比对方法。

### 检测模式

#### 模式A："等"与精确数量词冲突

| 定义段（定义方） | 指代段（引用方） | 矛盾 |
|:--|:--|:--|
| `包含A、B、C、D、E**等**在内的……` | `上述**五项**交付文件` | "等"=列举未尽 → 不应写死为"五项" |
| `主要功能包括X、Y、Z**等**` | `以上**三类**功能` | "等"=可能更多 → 不应锁定为"三类" |
| `涵盖……等多个方面` | `上述**七个**方面` | "多"=不确定数量 → 不应写死为"七个" |

**修复方案**：删除精确数量词 → "上述交付文件" / "以上功能" / "上述方面"。

#### 模式B：列举数量与宣称数量不一致

| 定义段 | 指代段 | 矛盾 |
|:--|:--|:--|
| `系统包含十大核心模块`（随后列出8个） | "十大模块"重复出现 | 宣称10但列举8 |
| `四大设计原则如下……`（列出5条） | "上述四大原则" | 宣称4但列举5 |

**修复方案**：使宣称数量与实际数量一致（修正宣称数或修正列举数）。

#### 模式C：中文数量词全量扫描

全文提取所有中文数量词（如"三项""五类""七个""八条""十二套"），逐条与所指数量的实际元素比对。

| 中文数量词 | 正则模式 |
|:--|:--|
| 基数+量词 | `[一两三四五六七八九十百千]+(项|类|个|条|套|种|层|级|步|环|方面|维度|模块|环节)` |

### 操作流程

```
1. 全文扫描所有"等"字列举段落（定义段列表）
2. 全文扫描所有"上述N项/类/个"指代句式（指代段列表）
3. 交叉比对：指代段是否指向定义段？定义段是否含"等"？
4. 对冲突项输出修复建议
5. 可选：启动模式C全量中文数量词审计
```

### P0级判定条件

- 定义段含"等"且指代段含精确数字 → **P1**（措辞矛盾，读者可自行推断但仍需修正）
- 宣称数量与列举数量不一致 → **P0**（硬伤，可直接导致审稿人质疑严谨性）

### 输出格式

```
跨段语义一致性审查报告
========================
定义段:
  L84: 包含质量证明书、材质单、检测报告、产品合格证、监检证**等** → 含"等"，数量不确定

指代段:
  L105: 上述**五项**交付文件 → 与L84"等"冲突

修复: L105 删除"五项" → "上述交付文件"
```

### 经验笔记

- 此类问题在交付包去重修复中常见：合并重复罗列时，为简洁而写出精确数量，忽略了原定义段的"等"字
- 审查工具是模式匹配而非语义理解，跨段语义矛盾不能用正则一步检出，必须建立专门的比对流程
- 中文"等"的语用含义有二：①列举未尽（"苹果、橘子等水果"）②列举煞尾（如公文用语），论文中通常为含义①

## #31 GB/T 7714引用重编号

### 背景

论文在多轮增删参考文献后，引用编号往往与正文首次出现顺序严重错乱——读者在第1章看到[3]时根本没出现过[2]，违反GB/T 7714顺序编码制的基本规则。修复此问题时，仅修改正文角标数字而不重新排列文献列表的物理段落顺序，会导致"角标指向[5]但参考文献表第5条是[5]的内容"看起来正确但实际错位的隐蔽错误。

### 检测方法

1. **提取正文全部引用**：按出现顺序扫描全文，记录每个引用角标及其首次出现的段落位置
2. **验证编号连续性**：首次出现顺序应为[1]→[2]→...→[n]，逐一比对
3. **验证文献列表物理顺序**：检查参考文献列表中段落的物理位置是否与编号一致

### 修复流程

```
1. 确定 old→new 映射表（按首次出现顺序分配新编号）
2. 正文全文替换所有角标：[old]→[new]
   注意：合并引用如[13,16]需整体替换为[3,4]
3. 参考文献表物理重排：
   a. 保存所有参考文献段落对象引用（复制后段落索引会变化）
   b. 逐段更新段落文本 [old]→[new]
   c. 从body中移除所有参考文献段落XML元素
   d. 按新编号顺序逐段 addnext() 重新插入
4. 验证：正文首次出现顺序 + 列表物理位置 + 编号连续性
```

### ⚠️ 二次映射陷阱（P0级操作错误）

**已重编号过一次的文件不能再套一次映射！**

- 症状：V3.3.15第一版错误——正文角标已从旧编号改为新编号，但参考文献列表物理段落顺序未改，导致读者看到[1]→[3]→[2]乱序
- 错误修复：在已重编号的V3.3.15上再次套用映射 → 二次错乱
- **正确做法**：从修改前的干净底本（V3.3.14）一次性完成角标替换+物理重排

### python-docx操作要点

| 操作 | 正确方法 | 错误方法 |
|------|---------|---------|
| 段落物理重排 | `body.remove(elem)` + `addnext()` 重新插入 | 仅修改段落文本中的数字 |
| 段落文本更新 | 清除所有`w:r`子元素 + 新增一个`w:r`含新文本 | 直接修改run.text（样式叠加） |
| 段落对象保存 | 在删除前保存`para_obj._element`引用 | 用段落索引（删除后索引漂移） |

### 输出格式

```
引用重编号报告
===============
Old→New映射:
  [1] → [1]  陈学东 高端压力容器 (Par12首引)
  [3] → [2]  特种设备安全法 (Par12，原[3]提前)
  [13] → [3]  阮斯洁 多维度工艺 (Par13首引)
  ...

修复结果:
  正文角标替换: 21处
  文献列表物理重排: 18条
  首次出现顺序: ✅ [1]→[18] 全部OK
  列表编号=物理位置: ✅ 18/18对齐
```

### P0级判定条件

- 引用编号不按首次出现顺序 → **P0硬伤**（违反GB/T 7714）
- 文献列表物理顺序与编号不一致 → **P0硬伤**（读者无法定位文献）
- 二次映射导致错乱 → **P0操作事故**（必须从干净底本重做）

### 经验笔记

- V3.3.9→V3.3.15实战：5条参考文献被修正/替换后，全文引用顺序完全打乱（[3]在[2]之前、[13,16]→[3,4]合并引用），需要全量重编号
- 合并引用格式如[13,16]转为[3,4]时，正则替换需匹配`[13,16]`整体而非逐个[13][16]替换
- 文献列表物理重排是最易出错的操作——python-docx在删除段落后段落索引立即变化，必须用`body.remove(elem)`操作XML元素而非段落索引
